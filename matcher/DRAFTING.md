# ClearPath — Compliance-Fact Grounding Layer (Phase 3a)

Phase 3 of ClearPath's roadmap (see `../STARTUP.md`) is "agentic, compliance-
aware application drafting for a matched job." **This document covers Phase
3a only: the fact-grounding/attribution layer that a future drafting step
must be built on top of.** No prose is generated by anything in this
document or in `draft_facts.py` — see "Phase 3b — not built yet" below for
what's still missing and why it's deliberately out of scope for this
increment.

## The problem this increment solves

Once `matcher/rank.py` (Phase 2) tells you *which* postings a specific
candidate can realistically pursue, the obvious next step looks like: feed
the candidate's resume, checker profile, and the matched posting into an LLM
and ask it to draft a cover letter or application blurb. **That's the wrong
order to build it in for a project whose whole value proposition is
compliance accuracy.** An LLM asked to write persuasively about a person's
eligibility will, left unconstrained, generalize past what's actually known
— e.g. turning a checker's honest `NOT_ENOUGH_INFO` on PGWP eligibility into
a confident "I am PGWP-eligible" sentence, because that's what a fluent
cover letter *sounds like*, not because any rule confirmed it. For a
compliance-adjacent tool, that's not a demo blemish, it's the exact failure
mode the product exists to prevent.

## "Attribute First, then Generate"

The named pattern for this situation — build and test the fact-retrieval and
attribution layer completely *before* any generation logic touches it, so
that attribution is structural rather than bolted on after the fact via a
prompt instruction or a post-hoc fact-check pass — is what this increment
implements. This run's research (`STARTUP.md`, 2026-07-22 entry) is what
identified it as the standard approach for exactly this situation
(compliance/eligibility-adjacent generation) rather than jumping straight to
"prompt an LLM and hope it doesn't overclaim."

Concretely: `assemble_fact_packet()` in `draft_facts.py` takes one
`matcher.rank.RankedResult` (a resume profile matched against one posting,
with the Phase 2 eligibility-compatibility gate already evaluated) and
produces a `FactPacket` — a structured, non-prose set of `allowed_claims`
and `forbidden_claims`, each one traced to a real source: a checker
`RuleResult`, the classifier's literal evidence spans, or the matcher's own
compatibility verdict. **A future Phase 3b drafting step would be
constrained to only ever assert what's in `allowed_claims`, and would be
checked (post-hoc, via an NLI-style claim-verification pass — also not built
in this increment) against never asserting anything in `forbidden_claims`.**
That constraint only works if this layer is itself correct and well-tested
first — which is the entire point of building it as its own increment.

## The `FactPacket` schema

```python
@dataclass
class Claim:
    text: str                                          # the statement itself
    source_type: Literal["checker", "classifier", "matcher"]
    citation: Optional[str] = None                      # a real checker/corpus id, or a literal
                                                          # evidence-span substring of the posting text
    rule_result_status: Optional[str] = None             # the checker Status value, or a verdict/
                                                          # confidence string when no RuleResult applies

@dataclass
class FactPacket:
    posting_id: str
    resume_profile_id: str
    allowed_claims: list[Claim]     # safe to state in a draft
    forbidden_claims: list[Claim]   # must NOT be stated, and why
    skill_overlap: list[str]        # resume skills that literally appear in the posting text
```

Every `allowed_claims` entry is guaranteed (by `packet_is_well_formed`) to
carry a non-null `citation` or `rule_result_status` — nothing is ever
asserted with no traceable source. This is the same discipline the
classifier's `evidence_spans` and the checker's `RuleResult.citations`
already follow elsewhere in the codebase, applied one level up.

### Where claims come from

- **Eligibility-category claims** (`_claims_for_category`): for each
  `CategoryCompatibility` already computed by `rank.py`'s
  `assess_compatibility()`, pull the *specific* checker `RuleResult`s
  relevant to that category (currently only `pgwp_track` maps to real
  per-profile `RuleResult`s — `pgwp_general_eligibility` and
  `pgwp_field_of_study`; see the judgment-call note in `draft_facts.py`'s
  `_CATEGORY_TO_RULE_NAMES` docstring for why `on_campus_only`,
  `co_op_exempt_eligible`, and `unrestricted` don't map to any RuleResult).
  A `COMPATIBLE` verdict → every relevant rule result is asserted `COMPLIANT`
  (a hard invariant, checked with an `assert`) → each becomes an allowed
  claim. An `UNCLEAR`/`INCOMPATIBLE` verdict → every non-compliant rule
  result becomes an explicit forbidden claim citing exactly why (the
  checker's own `Status` and `explanation`) — **this is the safety
  mechanism the whole layer exists for.**
- **Sponsorship claims** (`_claims_for_sponsorship`): derived from the
  Phase 0 classifier's `sponsorship_likelihood` + `evidence_spans`.
  `unlikely` → forbidden claim, citing the actual matched no-sponsor phrase.
  `likely` → an allowed claim ("may note openness to sponsoring"), citing
  the actual matched phrase — **plus, always, a second forbidden claim**
  guarding against overclaiming a *guaranteed* sponsorship, since the
  classifier is an honest heuristic (see `classifier/backend.py`'s own
  docstring — "not a calibrated probability"), never a verified employer
  confirmation, regardless of which way the signal leans.
- **`skill_overlap`** (`compute_skill_overlap`): a new, small, separately
  tested helper — no skill-overlap signal existed anywhere in the codebase
  before this increment. `rank.py`'s `compute_fit_scores` is whole-resume-
  text-vs-whole-posting-text embedding cosine similarity; it never
  identifies *which* specific skills overlap. This helper does literal,
  case-insensitive, word-boundary matching of each `resume.skills` entry
  against the posting text — deliberately not fuzzy or ML-based, so every
  entry in the list is a real, checkable substring of the posting, the same
  "no invented substrings" discipline `evidence_spans` already follows.

## Phase 3b — not built yet (read this before assuming more than what's here)

**This increment builds and tests the fact-grounding layer only. It does
not generate any prose, and it does not call any LLM.** The next increment
(Phase 3b, a separate future increment, not started) would:

1. Take a `FactPacket` and generate actual application/cover-letter prose,
   constrained to only assert what's in `allowed_claims` and never assert
   anything in `forbidden_claims`.
2. Require a **free-tier LLM API key** (Groq recommended over Gemini per
   this run's research — see `STARTUP.md`'s 2026-07-22 entry for the
   comparison) — no key has been requested or activated for this increment,
   per the standing operating constraint ("ask before using any API key,
   free-tier only") in `STARTUP.md` / `BUILD-STATUS.md`.
3. Add an **NLI-style (natural-language-inference) post-hoc claim-
   verification pass** — after generation, check every sentence in the
   draft against the `FactPacket` it was constrained to, flagging any
   sentence that isn't entailed by an `allowed_claims` entry. This has
   nothing to verify against yet in this increment (there's no generated
   prose), which is exactly why it comes after Phase 3a, not alongside it.

Until Phase 3b exists, this layer's only output is the structured
`FactPacket` itself — useful on its own as a "what can I honestly say about
myself for this job" checklist, but it is **not** an application drafter,
and this document does not claim otherwise.

## Scope and honest limitations

- Only `pgwp_track` maps to real per-profile checker `RuleResult`s; the
  other three eligible categories' claims are sourced from
  `CategoryCompatibility`'s own reason/citations, not from evaluating
  individual `RuleResult`s (see "Where claims come from" above and the
  judgment-call note in `draft_facts.py`).
- `skill_overlap` is literal word-boundary string matching, not semantic
  (won't match "ML" to "Machine Learning" as the same skill, for instance)
  — same "simple, traceable, no invented substrings" trade-off the
  classifier's `evidence_spans` already made.
- No LLM call, no prose generation, no persistence, no web/UI — same as
  every prior ClearPath module. See "Phase 3b — not built yet" above.
- Inherits every upstream module's own documented limitations unchanged
  (`classifier`'s adversarial-negation gap, `checker`'s no-CIP-code-lookup
  limitation, `matcher`'s no-live-postings gap) — this layer reads their
  output, it doesn't fix or mask any of it.
